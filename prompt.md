**Промпт для реализации прототипа на Replit.com**

## Описание проекта
Необходимо разработать рабочий прототип мобильного приложения (Kotlin) и бэкенда (Go) для персонализированного трекера здоровья и питания с использованием LLM (GigaChat) и интеграцией с сервисом доставки продуктов «Купер». Приложение должно позволять пользователю в естественной форме (через чат) вводить данные о питании, активности, сне и воде, получать интеллектуальную аналитику и рекомендации, а также формировать список покупок на основе выявленных дефицитов нутриентов.

Прототип должен быть развернут на Replit и демонстрировать ключевой функционал: обработку сообщений через GigaChat, анализ введенных данных, интеграцию с каталогом Купера и формирование рекомендаций.

## Технические требования
### Стек технологий
- **Мобильное приложение** – Kotlin (будет разрабатываться отдельно, но API должно быть готово; на Replit можно ограничиться веб-демо, имитирующим интерфейс)
- **Бэкенд** – Go (REST API)
- **База данных** – SQLite или PostgreSQL (для хранения пользователей, записей, истории)
- **Интеграции**:
  - GigaChat API (для обработки естественного языка, распознавания параметров, генерации рекомендаций)
  - API Купера (для получения списка продуктов, добавления в корзину) – предполагается наличие тестового доступа или заглушки

### Функциональные требования
#### 1. Базовый функционал
- **Естественный ввод данных через чат**  
  Пользователь может отправить сообщение типа: «Сегодня на завтрак съел овсянку с бананом и выпил стакан воды».  
  Бэкенд через GigaChat распознает: продукт (овсянка, банан), количество (примерное), воду (200 мл), активность (если есть), сон и т.д.  
  При необходимости LLM уточняет детали: «Сколько грамм овсянки?», «Вы занимались спортом?».  
  Реализовать быстрый переход к традиционной форме редактирования (можно просто кнопку "Ввести вручную").

- **Интеллектуальная оценка и рекомендации**  
  На основе введенных данных (возраст, вес, рост указываются при регистрации) LLM рассчитывает примерные калории, оценивает качество сна, гидратацию.  
  После тренировки пользователь может оценить самочувствие (опросник), на основе чего формируются рекомендации по восстановлению.

- **Аналитика и персональные инсайты**  
  Система выявляет недостающие данные (например, не введен ужин) и ненавязчиво запрашивает их.  
  На основе истории формируются советы по улучшению режима с ссылками на исследования (можно заглушки).

- **Гибкость интерфейса**  
  Пользователь может переключаться между чатом и формой детального ввода/просмотра. Данные синхронизируются.

#### 2. Epic: Анализ и коррекция рациона (User Stories)
- **US1: Выявление дефицита**  
  Бэкенд анализирует историю питания за последние 3 дня и выводит виджет с топ-1-2 дефицитами (например, «За последние 3 дня у вас дефицит клетчатки»).  
  *Критерий:* Виджет отображается в UI (веб-демо) с текстом дефицита.

- **US2: Персонализированные рекомендации товаров**  
  При нажатии на кнопку «Исправить» под виджетом дефицита система через API Купера подбирает 5-7 продуктов, богатых недостающим элементом.  
  Каждый продукт можно добавить в корзину (кнопка «+»). Реализована кнопка «Добавить все».  
  *Критерий:* Список продуктов загружается из Купера (или мок-данных) и отображается с ценой, изображением.

- **US3: «Вкусное решение» (Рецепт)**  
  Под списком продуктов показывается 1-2 карточки рецепта, которые помогают закрыть дефицит (например, салат с авокадо).  
  При нажатии на рецепт все недостающие ингредиенты (кроме базовых) автоматически подсвечиваются и добавляются в корзину.  
  *Критерий:* Рецепт генерируется на основе имеющихся продуктов (можно через GigaChat или предопределенный список).

- **US4: Формирование шопинг-листа**  
  Отдельный раздел «Список покупок на неделю», который автоматически формируется на основе целей пользователя и предпочтений (например, для набора мышечной массы или похудения).  
  Список группируется по категориям (овощи, молочка и т.д.), отображается итоговая сумма.  
  Возможность удалить позицию из списка.  
  *Критерий:* Список можно отредактировать и отправить в корзину Купера.

## Интеграции
### GigaChat
- Использовать API GigaChat (необходимо получить ключ).  
  Запросы к LLM должны быть оптимизированы для извлечения структурированных данных из свободного текста, а также для генерации рекомендаций.  
  Пример промпта: «Извлеки из сообщения информацию о питании, воде, сне и активности. Верни JSON с полями: meal (завтрак/обед/ужин/перекус), продукты (список с названиями и примерным весом), water (мл), sleep (часы), activity (тип, длительность). Если данных недостаточно, задай уточняющий вопрос».

### Купер
- API Купера (если нет доступа, использовать мок-данные).  
  Необходимо получать список продуктов по категориям (например, «продукты богатые клетчаткой»), добавлять товары в корзину (имитация).

## Дизайн
Приложение должно следовать современным мобильным тенденциям (Material Design). Основные экраны:
- Чат (список сообщений, поле ввода, быстрые кнопки для ввода активности/воды/сна)
- Дашборд с виджетами: дефициты, рекомендации, краткая статистика (калории, вода, сон)
- Экран списка покупок
- Форма детального ввода (традиционная)

Референсы дизайна приложены (предположительно скриншоты). На Replit достаточно реализовать адаптивную веб-версию, имитирующую мобильный интерфейс (можно использовать CSS-фреймворк типа Bootstrap или Tailwind).

## Требования к прототипу на Replit
- **Бэкенд**: Go с фреймворком (например, Gin или Echo). Реализовать REST API:
  - `POST /api/chat` – принимает сообщение, обрабатывает через GigaChat, возвращает ответ (текст и обновленные данные)
  - `GET /api/user/{id}/deficits` – возвращает текущие дефициты
  - `GET /api/products?deficit=...` – получает продукты из Купера по дефициту
  - `POST /api/cart/add` – добавить товар в корзину (заглушка)
  - `GET /api/shopping-list` – сформировать список на неделю
  - и др.
- **База данных**: SQLite (для простоты) с таблицами users, food_entries, activities, sleep, deficits и т.д.
- **Веб-интерфейс**: HTML/CSS/JS (можно использовать React/Vue для демонстрации, но не обязательно). Главное – показать взаимодействие с бэкендом.
- **Конфигурация**: через переменные окружения (GIGACHAT_API_KEY, KUPER_API_KEY и т.д.)

## Критерии приемки
- [ ] Пользователь может отправить сообщение в чат, и бэкенд возвращает структурированный ответ (распознанные данные).
- [ ] На дашборде отображается виджет с дефицитом (если есть за последние 3 дня).
- [ ] При клике «Исправить» подгружается список продуктов (из Купера или мока).
- [ ] Есть возможность добавить продукты в корзину.
- [ ] Отображается рецепт, ингредиенты можно добавить в корзину.
- [ ] Раздел «Список покупок на неделю» формируется (можно по умолчанию) и доступен для редактирования.
- [ ] Код размещен на Replit, инструкция по запуску (включая установку зависимостей Go и настройку переменных) описана в README.

## Дополнительно
- Поскольку GigaChat может быть платным или требовать ключ, можно предусмотреть режим демо без реального API (имитация ответов), но с возможностью легко переключить на реальный ключ.
- Интеграцию с Купером также можно замокать (например, статический JSON), но структура должна соответствовать реальному API.

**Результат:** Рабочий прототип на Replit, который демонстрирует всю цепочку – от ввода сообщения до формирования списка покупок с учетом дефицитов и рецептов. Бэкенд готов к интеграции с мобильным приложением на Kotlin.
